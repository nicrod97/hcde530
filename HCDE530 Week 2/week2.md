# Week 2 — Competency 2: Code literacy and documentation

## What I practiced this week

This week I focused on **code literacy**: how scripts are organized, how data moves through them, and how that connects to clear documentation.

## What improved

**Understanding structure.** I got better at three things in particular:

1. **Functions** — Finding a named block of code and asking what job it does in the overall story.
2. **Loops** — Seeing how the program walks through a list (for example, row by row in a file) and repeats the same small steps.
3. **Input and output** — Following where data comes in (for example, a CSV), what happens in the middle (counts, summaries), and what gets printed or saved at the end.

That kind of tracing matters when you are not trying to be an engineer but you *do* need to trust and explain what the code did.

## How I documented my work (Competency 2)

I supported my own learning and others who might read the repo by:

- **Commit messages** — Short notes that capture *why* a change happened, not only *what* changed.
- **Comments in code** — Light explanations next to the parts that are easy to misread on first pass.
- **Notes on terminal commands** — Writing down the exact commands and folder I used so I can repeat a successful run without guessing.

## What was hardest

The trickiest part was understanding **the environment**: which folder the terminal was “in” and how that affected file paths and errors. Once that clicked, a lot of “mysterious” failures made sense as location or path problems, not magic.

## How this connects to UX research

Two links feel strongest:

1. **Communicating steps to teammates** — Research is clearer when you can describe processing steps in plain language and point to repeatable commands or scripts, the same way you might document a coding or interview protocol.
2. **Understanding the engineering process** — Reading structured code is closer to reading a method section: you see assumptions, inputs, and outputs. That helps when you collaborate with engineers and want alignment on what “counts” as clean, auditable work.
